# 科研工程工作流 (Research Engineering Workflow)

帮助研究者完整走通"文献调研 → 技术选型 → 方案设计 → 实验验证 → 成果交付"这条路。

## 核心原则

**所有跟用户交互和展示的文字必须用中文。** 代码注释、git commit 可以用英文。技术术语保留英文原文并首次出现时附中文解释。

## 能力模块一：文献消化

### 什么时候用
- 用户给论文 PDF、技术博客
- 需要理解新领域技术全景
- 技术选型前的前置调研

### 怎么做
对每份材料提取：

1. **一句话总结** - 他们实际构建了什么
2. **架构图** - ASCII art 画出 pipeline
3. **关键数据** - 提取 3-5 个最相关的量化结果
4. **跟我们的关系** - 可以借鉴什么？不能直接迁移的局限性

### 产出格式

```markdown
# [项目名] — 参考材料消化汇总

> **目的**：[一句话]

## 📋 材料清单
| # | 材料 | 类型 | 核心内容 |

## [每份材料]
### 标题
> **论文/来源**: 完整引用
#### 它做了什么
[ASCII pipeline 图]
#### 关键数据
[数据对比表]
#### 对我们的启发
> [!IMPORTANT] [关键洞察]

## 横向对比：所有方法一览
[跨材料综合对比表]
```

## 能力模块二：技术地图

### 怎么做
1. **拆解系统**为正交的阶段（如：编码 → 检索 → 对齐 → 组装）
2. **每个阶段**列出所有可行技术，做多维度对比
3. **标注依赖关系**
4. **区分已知和未知**

用状态标志：✅ 已验证、⚠️ 需实测、❌ 已排除、🔥 有潜力但未验证
用来源标记：🔬 实测数据、📖 论文数据、📐 推算/外推

## 能力模块三：方案设计

1. **设计 2-4 套完整方案**，各有侧重（如：极速派、精度派、务实派、混合派）
2. **每套方案**包含：各环节工具选择、ASCII 架构图、定量估算、优劣势风险
3. **做对比矩阵**，让决策者看一张表就能选方案
4. **给推荐**，附明确理由

每个估算都需要依据。没有出处的数字是小说。

## 能力模块四：实验验证 (Benchmarking)

1. **找出关键未知** - 哪些假设如果错了，会改变推荐？
2. **设计最小实验** - 测什么、配置是什么、什么结果会改变决策？
3. **系统地收集数据**
4. **产出报告**，带可视化

每次只变一个变量。每个结果都要有 "so what"。

## 能力模块五：成果交付

### 总结文档
假设读者是聪明的领域专家，但没读过你读的论文，也没碰过你的代码。用大白话说清目标和大图。

### 进度跟踪
维护活的任务清单：✅ 完成、🔄 进行中、⏳ 阻塞、❌ 放弃

## 反模式

1. **"论文崇拜"** - 发在顶刊 ≠ 在你的场景下 work
2. **"过早优化"** - 先跑通端到端 pipeline，再优化瓶颈
3. **"选择困难症"** - 挑最可能的 2-3 个方案，测了再说
4. **"无源估算"** - 每个数字都要有出处
5. **"铁板一块"** - 架构每个阶段要能独立替换
